## [CRITICAL #1] Timing Attack w Porównaniu API Key

**[PLIK]** `backend_app/utils/dependencies.py:16`

**[PROBLEM]**
Porównanie API key podatne na timing attack:
```python
def require_api_key(request: Request) -> None:
    if not settings.api_key:
        return
    key = request.headers.get("X-API-Key")
    if not key:
        auth = request.headers.get("Authorization", "")
        if auth.startswith("Bearer "):
            key = auth[7:]
    if key != settings.api_key:  # ← TIMING ATTACK!
        raise HTTPException(status_code=401, detail="Invalid or missing API key")
```

Operator `!=` to porównanie lexicographiczne, które trwa różnie w zależności od liczby pasujących znaków na początkach stringów. Atakujący może mierzyć czas odpowiedzi HTTP aby iteracyjnie odgadywać znaki klucza.

**[RYZYKO]**
- Potencjalny wyciek API key przez time-based side-channel attack
- Jeśli klucz wycieknie, atakujący może się uwierzytelniać bez ograniczeń
- Krytyczne dla bezpieczeństwa systemu w produkcji

**[FIX]**
Zastąp niezabezpieczonym porównaniem na constant-time comparison:

```python
import secrets
from fastapi import Depends, HTTPException, Request

def require_api_key(request: Request) -> None:
    if not settings.api_key:
        return

    key = request.headers.get("X-API-Key")
    if not key:
        auth = request.headers.get("Authorization", "")
        if auth.startswith("Bearer "):
            key = auth[7:]

    # ZMIANA: użyj secrets.compare_digest dla constant-time comparison
    if not secrets.compare_digest(key or "", settings.api_key or ""):
        raise HTTPException(status_code=401, detail="Invalid or missing API key")
```

**[EVIDENCJA]**
- Python docs: https://docs.python.org/3/library/secrets.html#secrets.compare_digest
- OWASP: Timing Attacks

```python
def resolve_path(fs_root: str, user_path: str) -> str:
    """Zamienia user_path na realpath i sprawdza, czy jest pod fs_root."""
    base_root = os.path.realpath(fs_root)  # /media/lobo/BACKUP/KlimtechRAG
    raw_path = os.path.expanduser(user_path or ".")

    if os.path.isabs(raw_path):
        candidate = os.path.realpath(raw_path)  # Rozwiązuje symlinki
    else:
        candidate = os.path.realpath(os.path.join(base_root, raw_path))

    if not candidate.startswith(base_root):  # ← SECURITY CHECK
        raise FsSecurityError("Path not allowed")
    return candidate
```

## [MEDIUM #3] Brak Input Validation na `max_file_size_bytes` w Upload

**[PLIK]** `backend_app/routes/ingest.py:301-395`

**[PROBLEM]**
Endpoint `/upload` nie sprawdza rozmiaru pliku PRZED zapisem na dysk:

```python
@router.post("/upload")

    req: Request,
    request_id: str = Depends(get_request_id),
):
    require_api_key(req)
    apply_rate_limit(get_client_id(req))
```

**[RYZYKO]**

- settings.max_file_size_bytes (200MB) nie jest egzekwowany
- Disk exhaustion

**[FIX]**

```python
@router.post("/upload")
async def upload_file(file: UploadFile, ...):
    require_api_key(req)

    filename = file.filename
    ext = os.path.splitext(filename)[1].lower()

    if ext not in settings.allowed_extensions_docs:
        raise HTTPException(status_code=400, detail=f"Extension not allowed: {ext}")

    # ZMIANA: Czytaj plik w chunkach i sprawdź rozmiar
    upload_dir = os.path.join(settings.upload_base, EXT_TO_DIR.get(ext, "other"))
    os.makedirs(upload_dir, exist_ok=True)

    file_path = os.path.join(upload_dir, filename)
    total_bytes = 0
    max_bytes = settings.max_file_size_bytes

    with open(file_path, "wb") as f:
        while chunk := await file.read(8192):
            total_bytes += len(chunk)
            if total_bytes > max_bytes:
                os.unlink(file_path)  # Usuń niedokończony plik
                raise HTTPException(
                    status_code=413,
                    detail=f"File too large (max: {max_bytes / 1024 / 1024:.1f}MB)"
```

```python
class SubprocessTimeout(Exception):
    """Custom exception dla subprocess timeouts."""
    pass

def run_with_timeout(cmd: list, timeout: int = 30) -> subprocess.CompletedProcess:
    """Uruchamia process z timeout-em i spójnym error handling."""
    try:
        result = subprocess.run(
            cmd,
            capture_output=True,
            text=True,
            timeout=timeout,
        )
        if result.returncode != 0:
            logger.warning(f"Process failed: {' '.join(cmd)}, stderr: {result.stderr[:200]}")
        return result
    except subprocess.TimeoutExpired as e:
        logger.error(f"Subprocess timeout: {' '.join(cmd)}")
        raise SubprocessTimeout(f"Command timed out after {timeout}s")
```

## [INFO #1] Cache Bez TTL Limit w Chat Responses

**[PLIK]** `backend_app/routes/chat.py:34-59`

**[UWAGA]**
Cache `_answer_cache` ma TTL (1 godzina) i max size (500), ale może rosnąć bez limitu jeśli TTL się nie wyczyści:

```python
_answer_cache: Dict[str, Tuple[str, float]] = {}
CACHE_TTL = 3600
CACHE_MAX_SIZE = 500

def get_cached(query: str) -> Optional[str]:
    if query in _answer_cache:
        answer, ts = _answer_cache[query]
        if time.time() - ts < CACHE_TTL:
            return answer
        del _answer_cache[query]  # ← Usuwanie LAZY!
    return None

def set_cached(query: str, answer: str) -> None:
    if len(_answer_cache) >= CACHE_MAX_SIZE:
        oldest = min(_answer_cache, key=lambda k: _answer_cache[k][1])
        del _answer_cache[oldest]  # ← FIFO when full
    _answer_cache[query] = (answer, time.time())
```

**[OBSERWACJA]**
- Cache jest in-memory (GC je wyczyści przy reload)
- LAZY cleanup — stare wpisy mogą wisieć do CACHE_MAX_SIZE
- Dla 1 godziny TTL i 500 size — rozsądnie dla dev

**[REKOMENDACJA]**
W produkcji używaj Redis lub implementuj active cleanup task.
