## 3. Znane błędy i problemy

### 🔴 Krytyczne

1. **`ingest_gpu.py` zabija `start_klimtech.py`**
   - Opis: Gdy `start_klimtech.py` działa, uruchomienie `ingest_gpu.py` zatrzymuje LLM i backend
   - Efekt: `start_klimtech.py` kończy się ("Wszystkie procesy zakończone")
   - Obejście: Używać `start_backend_gpu.sh` osobno lub nie uruchamiać `ingest_gpu.py` gdy `start_klimtech.py` działa

2. **Backend po restarcie przez `ingest_gpu.py` działa na CPU dla chat**
   - Opis: Po zakończeniu indeksowania i przywróceniu LLM, embedding wraca na CPU
   - Przyczyna: `start_klimtech.py` uruchamia backend z `KLIMTECH_EMBEDDING_DEVICE=cpu`
   - Rozwiązanie: Zmienić w `start_klimtech.py` na `cuda:0` jeśli LLM nie używa całego VRAM

### 🟡 Średnie

3. **`stop_klimtech.py` nie zabija procesów do końca**
   - Backend i watchdog mogą zostać uruchomione po zatrzymaniu
   - **Rozwiązanie ręczne:**
   ```bash
   pkill -f "uvicorn backend_app" 2>/dev/null; fuser -k 8000/tcp 2>/dev/null; sleep 2; ss -tlnp | grep 8000 || echo "Port 8000 wolny"
   ```

4. **Wiele instancji backendu**
   - Po zatrzymaniu `start_klimtech.py` i uruchomieniu `start_backend_gpu.sh` mogą zostać stare procesy
   - Rozwiązanie: `pkill -f backend_app.main` przed uruchomieniem

4. **`@router.on_event("startup")` deprecated**
   - W `admin.py` używany stary sposób eventów
   - Rozwiązanie: Przenieść do `lifespan` w `main.py`

### 🟢 Niskie

5. **LSP errors w config.py**
   - `AnyHttpUrl` type mismatch
   - Nie wpływa na działanie

## 5. Sugerowany workflow

### Opcja A: Rozdzielony (zalecane)
```bash
# Terminal 1: Tylko LLM
./start_klimtech.py  # (ale bez backendu)

# Terminal 2: Backend z GPU
./start_backend_gpu.sh

# Gdy trzeba zindeksować duże pliki:
python backend_app/scripts/ingest_gpu.py --limit 10
```

### Opcja B: Zintegrowany
```bash
# Normalny tryb (embedding na CPU)
./start_klimtech.py

# Gdy trzeba zindeksować (zabije start_klimtech.py!)
python backend_app/scripts/ingest_gpu.py --limit 10

# Należy ręcznie zrestartować:
./start_klimtech.py
```
